# プログラミング未経験でもAIと作れる！自分だけのWebアプリ開発ガイド

## はじめに

プログラミング経験ほぼゼロの自分が、AIに全部作ってもらって2つのWebアプリを公開した。

ひとつは Life OS。日記、ToDo、読書記録、AIチャット、カレンダー連携、歩数記録、週次レポートが入った個人ダッシュボード。

もうひとつは Project OS。趣味の開発プロジェクトや資格勉強を管理するアプリ。AIがタスクを提案してくれて、ガントチャートでスケジュールが見える。GitHubのコミット履歴から進捗を自動検知する機能までついてる。

ランニングコストはほぼゼロ。Gemini APIの利用料が2日で31円。月に換算しても500円いかない。

「バイブコーディング」という言葉を聞いたことがあるだろうか。AIに自然言語で指示してコードを書いてもらう開発スタイルのこと。自分がやったのはまさにこれで、プログラミングの知識がなくても、日本語で「こういうの作って」と言えばアプリが生まれる。

この記事では、自分が実際にどうやって2つのアプリを作ったか、全体の流れとツールの使い方を書く。

## 全体の流れ：5ステップで回る

最初に全体像を見せる。アプリ開発と聞くと身構えるかもしれないけど、やってることはこの5ステップの繰り返し。

```
① アイデア・設計（Claude チャット）
    ↓  「こういうの作りたい」→ 指示書ができる
② 実装（Claude Code）
    ↓  指示書を渡す → コードが生まれる
③ コード管理（GitHub）
    ↓  保存・バックアップ
④ 公開（Cloudflare）
    ↓  世界中からアクセスできる
⑤ 使ってみる → 「〜もつけたい」 → ①に戻る
```

自分のLife OSは、このサイクルを何周もした結果、ひとこと日記がAIコーチ付き個人ダッシュボードに育った。Project OSは別の「作りたい」が生まれて、同じサイクルでゼロから作った。

## 各ツールの役割

4つのツールを使う。全部無料。たとえ話で説明する。

**Claude（チャット）** は設計士。「こういう家がほしい」と相談すると、間取りや構造を提案してくれる。「予算はゼロで」「部屋は3つで」みたいな要望を伝えると、図面（指示書）を引いてくれる。

**Claude Code** は大工。設計士が引いた図面をもとに、実際に家を建ててくれる。釘を打つのも壁を塗るのも全部やってくれる。自分は横で「そこはもうちょっと右」って言うだけ。

**GitHub** は倉庫。建材（コード）を保管する場所。いつ何を変えたか全部記録されてるから、「先週のバージョンに戻したい」もできる。

**Cloudflare** は土地と住所。建てた家をインターネット上に設置して、URLを発行してくれる。スマホからアクセスすれば、自分のアプリがいつでもどこでも使える。

この4つを順番に使っていく。プログラミングの知識は要らない。日本語で指示するだけ。

## 「具体的にどうやるの？」

ここから先は、実際の手順・画面・コツを全部見せます。

## ステップ0：準備するもの

必要なアカウントは5つ。全部無料で作れる。

**Googleアカウント** ── 持ってる人がほとんどだと思う。Gemini APIの取得とGoogleカレンダー連携に使う。

**GitHubアカウント** ── https://github.com/ で登録。コードの保存場所。

**Cloudflareアカウント** ── https://www.cloudflare.com/ で登録。アプリの公開場所。

**Anthropicアカウント** ── https://www.anthropic.com/ で登録。Claude（チャット）とClaude Code（デスクトップアプリ）を使うため。Claude Proプラン（月$20）に入るとClaude Codeが使える。

**Google Cloud Console** ── Gemini APIキーの取得場所。Googleカレンダー連携にも使う。次のセクションで詳しく説明する。

## Gemini APIキーの取得

ここ、意外と迷う人が多いポイント。Gemini APIキーの取得方法は2つある。Google AI Studio版とGoogle Cloud Platform（GCP）版。

### AI Studio版（手軽さ重視）

https://aistudio.google.com/ にGoogleアカウントでログイン。画面上の「Get API key」を押す。以上。30秒で完了。クレジットカードの登録は不要。

無料枠は1日あたり250リクエスト（Gemini 2.5 Flash）。個人の日記アプリやプロジェクト管理で使う程度なら、無料枠に余裕で収まる。超えた分は従量課金だけど、個人開発なら月に数百円で済む。

### GCP版（自分はこっちを使ってる）

https://console.cloud.google.com/ からプロジェクトを作成して、Gemini APIを有効化して、APIキーを発行する。手順は多いけれど、後述するGoogleカレンダー連携（OAuth設定）でもGCPのプロジェクトを使うので、最初からGCPで統一しておくと管理が楽。自分も実際にGCP版を使っている。

$300分の無料トライアル（90日間）がつくが、個人開発のテキストAPI利用で使い切ることはまずない。クレジットカード登録が必要だが、無料トライアルが終わっても勝手に課金されることはない（手動で有料アカウントに切り替えが必要）。

### 2つの方法の比較

|  | AI Studio版 | GCP版 |
|---|---|---|
| 取得の手軽さ | ◎ 30秒、ワンクリック | △ 10分、複数ステップ |
| クレカ登録 | 不要 | 必要 |
| 無料枠 | 250リクエスト/日 | $300/90日トライアル |
| 超過時の料金 | 従量課金（激安） | 従量課金（同額） |
| APIの中身 | まったく同じ | まったく同じ |
| データの扱い | ⚠ 無料枠はトレーニングに利用される可能性あり | ◎ トレーニングに利用されない |
| APIキーの制限設定 | × できない | ◎ IPアドレス制限、リファラー制限、API単位の制限が可能 |
| 使用量の監視 | 簡易的 | ◎ 詳細なダッシュボード、アラート設定 |
| 他のGoogle API連携 | 別途GCPが必要 | ◎ 同じプロジェクトで一元管理 |
| 向いている人 | とにかく早く試したい人 | 日記など個人データを扱う人、OAuth連携も使う人 |

大事なのは、返ってくるAPIの中身は同じということ。入口が違うだけ。ただし **データの扱いには大きな差がある。**

### データプライバシーの話（ここ大事）

AI Studio版の無料枠で送ったプロンプトと応答は、Googleがモデル改善やトレーニングに使用する可能性がある。しかもこれはオプトアウト（拒否）できない。課金を有効にすれば「有料サービス」扱いになりトレーニングには使われなくなるが、無料のまま使う場合は覚悟が必要。

一方、GCP版（Vertex AI経由）は明確に「ユーザーの許可なくデータをトレーニングに使用しない」と宣言している。ゼロデータリテンション（データを保持しない）オプションも用意されている。

日記やToDo、体調記録のような個人的なデータをAIに送る場合、これは無視できないポイント。自分がGCP版を選んだのもこれが理由。日記の内容がGoogleのAI学習に使われるかもしれない、と思うと気持ちよく書けない。

なお、どちらの方式でも不正利用検知のために最大55日間プロンプトが保持される。これはオプトアウトできない。ただしこれはトレーニング目的ではなく、ポリシー違反の検出専用と明記されている。

### セキュリティの観点

GCP版にはもうひとつ優位性がある。APIキーの制限設定。AI Studio版のAPIキーは制限なしの「裸のキー」で、万が一漏洩した場合に第三者が無制限に使えてしまう。GCP版なら、APIキーに「このドメインからのリクエストのみ許可」「このIPアドレスのみ許可」「Gemini APIのみ使用可能」といった制限をかけられる。キーが漏れても被害を最小限に抑えられる。

また、GCPではAPIの使用量をリアルタイムで監視できる。「1日のリクエスト数が100を超えたらメール通知」のようなアラートも設定可能。不正利用の早期発見に役立つ。

### 結局どっちがいいの？

「とりあえず触ってみたい」「まだ何を作るか決まってない」ならAI Studio版でいい。30秒で始められるのは正義。

ただし、日記アプリのように個人的なデータを扱うなら、最初からGCP版をおすすめする。自分もGCPを使っている。手順は多いけど一度やれば終わりだし、Googleカレンダー連携をするときにどのみちGCPのプロジェクトが必要になる。APIキーを環境変数で管理していれば、AI Studio版から始めて後からGCPに差し替えることもできる。

## ステップ1：Claudeとアイデア出し・設計

準備ができたら、いよいよアプリを作り始める。

ただし、いきなりClaude Codeを起動しない。まずはClaudeのチャットで相談する。

Claude Code（大工）にいきなり「家建てて」と言っても、どんな家が欲しいか伝わらなければ意図と違うものができあがる。先に設計士（Claudeチャット）と相談して図面を作る。この段階が実は一番大事。

Claudeのチャット（Webブラウザ https://claude.ai/ またはデスクトップアプリ）を開いて、作りたいものを話す。

自分の場合、Life OSの最初はこんな会話だった。

> 自分: 日記アプリ作りたい。スマホでサクッと記録できて、データはちゃんと構造化しておきたい。将来AIに分析させたいから。
>
> Claude: 技術スタックはどうしますか？候補としてはReact + Supabase、Next.js + Vercel……
>
> 自分: 無料がいい。サーバー代払いたくない。
>
> Claude: それならCloudflare Workers + D1 + Pagesがおすすめです。全部無料枠で収まります。構成は……
>
> 自分: スプレッドシートにデータ入れたいんだけど。
>
> Claude: GAS + Spreadsheetで始めて、速度が必要になったらCloudflareに移行する手もあります。

こんなふうに、要望を伝えるとClaudeが選択肢を出してくれる。「これはいる」「これはいらない」「予算はゼロで」と返していくと、自分の要望に合った設計がだんだん固まっていく。

この対話の結果として指示書ができあがる。指示書とは、アプリの設計図みたいなもの。データベースの構成、APIの一覧、画面のレイアウト、デザインの方針がまとまったドキュメント。

Project OSの指示書では、こんな内容を決めた。

* データベースのテーブル（projects, tasks, recurring, settingsなど）
* 各テーブルのカラム（プロジェクト名、タスク名、期限、ステータスなど）
* APIのエンドポイント（プロジェクト作成、タスク追加、AI提案など）
* 画面構成（ダッシュボード、プロジェクト一覧、ガントチャート、設定）
* AIキャラ「ピアちゃん」の性格設定
* デザインのカラーパレット

最初から完璧な指示書を作る必要はない。最小限の機能で始めて、使いながら「〜もつけたい」を繰り返せばいい。自分のLife OSも最初は「絵文字で気分を選んで一言書いて保存」だけだった。そこからToDo、読書記録、AIチャット、カレンダー連携……と、指示書を更新しながら育てていった。

## ステップ2：Claude Codeで実装

指示書ができたら、Claude Codeの出番。

Claude Codeのデスクトップアプリを起動して、作業フォルダを開く。ステップ1で作った指示書をそのまま貼り付ける。

すると、Claude Codeがフォルダ構成を作り、ファイルを生成し、コードを書き始める。画面の中でリアルタイムにファイルが生まれていく。数分〜数十分で、動くアプリの骨組みができあがる。

自分がやることは、Claude Codeの作業を見守ることと、途中で聞かれたら答えること。「このフォルダにファイルを作っていい？」「この機能はこういう仕様で合ってる？」みたいな確認が来ることがある。

**エラーが出たら？**

出る。頻繁に出る。でも慌てなくていい。Claude Codeにそのまま「エラー出た」と伝えればいい。エラーメッセージをコピペする必要すらないことが多い。Claude Codeが自分でログを読んで、原因を特定して、修正してくれる。

**途中で方針を変えたくなったら？**

「やっぱりこの画面にボタン追加して」「色をミントグリーンに変えて」と日本語で言えばいい。Claude Codeが該当ファイルを見つけて修正してくれる。

**指示のコツがある。**

曖昧な指示だと意図と違うものができる。「いい感じにして」より「背景色を#F0FAF6にして、ボタンは角丸12pxで高さ44px」のほうが正確に伝わる。デザインの参考画像があれば「この画面みたいにして」と見せるのも効果的。

逆に、最初の大枠は曖昧でいい。「日記アプリ作って」から始めて、動いたものを見ながら「ここをこうして」と具体的に修正していく。このやり取りの往復がバイブコーディングの本質。

## ステップ3：GitHubにプッシュ

コードが動いたら、GitHubに保存する。

これもClaude Codeにお願いできる。

> 自分: GitHubにリポジトリ作ってプッシュして

Claude Codeがgitコマンドを実行して、リポジトリの作成からコードのアップロードまでやってくれる。GitHub CLIが設定されていれば、ブラウザを開く必要すらない。

GitHubに保存する意味は3つ。

ひとつはバックアップ。パソコンが壊れてもコードはクラウドに残ってる。

ふたつめは変更履歴。いつ、何を、どう変えたかが全部記録される。「3日前のバージョンに戻したい」ができる。

みっつめはCloudflareとの連携。GitHubにコードを保存すると、Cloudflareが自動でアプリを更新してくれる。これが次のステップ。

## ステップ4：Cloudflareで公開

アプリをインターネットに公開する。初回だけCloudflareの管理画面で設定が必要。

**Cloudflare Pagesの作成** ── Cloudflareダッシュボードにログインして「Workers & Pages」→「Create」→「Pages」→「Connect to Git」でGitHubのリポジトリを選択。ビルド設定を入れてデプロイ。

**D1データベースの作成** ── 「Workers & Pages」→「D1」→「Create database」でデータベースを作る。名前をつけるだけ。

**環境変数の設定** ── Workersの設定画面で、Gemini APIキーやGitHubトークンを環境変数として登録する。コードに直接書くとGitHubに公開されてしまうので、必ず環境変数を使う。

ここは管理画面が英語だったり、設定項目が多かったりして、一番とまどうポイントかもしれない。でも一度やれば終わり。わからなければClaude（チャット）に「Cloudflare Pagesの設定方法を教えて」と聞けばステップごとにガイドしてくれる。

一度接続すると、以降は全自動。 GitHubにプッシュするだけで、数十秒後に本番環境に反映される。つまり、Claude Codeで修正→「GitHubにプッシュして」→自動で公開。このスピード感は一度体験すると戻れなくなる。

## ステップ5：使う→改善→繰り返し

アプリが公開されたら、実際に使ってみる。

使い始めると必ず「ここが使いにくい」「この機能もほしい」が出てくる。自分の場合はこうだった。

日記を書き始めたら「ToDoも一緒に管理したい」。
ToDoを足したら「読書記録もほしい」。
データが溜まってきたら「AIに分析してほしい」。
AIをつけたら「キャラをつけたらもっと楽しくない？」。

ひとつ足すと、もうひとつほしくなる。それを毎回、Claudeチャットで相談→指示書を更新→Claude Codeで実装→GitHubプッシュ→自動デプロイ、のサイクルで回す。

Life OSはこのサイクルを回し続けた結果、ひとこと日記が日記＋ToDo＋読書記録＋AIチャット＋カレンダー連携＋フィットネス＋週次レポートの個人ダッシュボードに育った。

さらに「趣味プロジェクトの管理もしたい」という別の欲が生まれて、同じサイクルでProject OSをゼロから作った。AIによるタスク提案、スケジュール逆算、ガントチャート、GitHub連携、学習ロードマップ。全部Claudeに相談して、Claude Codeに作ってもらった。

最初から全部入りを目指さなくていい。 小さく作って、使って、育てる。この繰り返しがバイブコーディングの醍醐味。

## 発展編：外部ツールとの連携

アプリが動き始めると「あのサービスのデータも取り込みたい」が出てくる。Googleカレンダーの予定を表示したい、歩数を自動で取り込みたい、Xに共有したい。

連携先によって難易度が全然違うので、3段階に分けて説明する。

### レベル1：APIキーだけで使える（簡単）

一番楽なパターン。APIキーをCloudflare Workersの環境変数に入れて、Claude Codeに「このAPIと連携して」と言えばコードを書いてくれる。

自分のアプリで使ったレベル1の連携はこのあたり。

**Gemini API** はアプリのAI機能の心臓部。ピアちゃんのチャット、タスク提案、週次レポートのコメント生成、全部これ。

**OpenBDと国立国会図書館API** はLife OSの読書記録で使ってる。ISBNで検索するとOpenBDから、タイトルで検索すると国立国会図書館APIから書籍情報が返ってくる。どちらも完全無料、利用登録すら不要。

**GitHub API** はProject OSでコミット履歴を取得するのに使ってる。Personal Access Tokenをヘッダーに付けるだけ。

**X（Twitter）への共有** もレベル1。APIを使わず、インテントURLという仕組みでXの投稿画面にテキストを渡すだけ。日記の履歴画面で「Xに共有」ボタンを押すと、投稿画面がテキスト入りで開く。ユーザーが自分で「ポストする」を押すまで投稿されないので事故もない。

**Google Fitの歩数データ** もレベル1。Google FitからエクスポートしたCSVファイルをアプリにインポートする方式。OAuthの設定は不要で、CSVをアップロードするだけ。

### レベル2：OAuth認証が必要（ちょっと大変）

Googleカレンダー、Spotify、LINE、Slackなど。「ユーザーの代わりにデータを取得する」ので、APIキーだけじゃなくてOAuth（オーオース）という認証の仕組みが必要になる。

Googleカレンダー連携を例に、ざっくりした流れを説明する。

まずGoogle Cloud Console（https://console.cloud.google.com/）でOAuthの設定をする。プロジェクトを作って、Google Calendar APIを有効化して、OAuth同意画面を設定して、クライアントIDとシークレットを取得する。Gemini APIでGCP版を使ってる人はプロジェクトが既にあるから、そこにAPIを追加するだけ。

次にCloudflare Workers側でOAuthのフローを実装する。ユーザーがアプリ内の「Googleカレンダーを連携」ボタンを押す→Googleのログイン画面が開く→許可する→アプリにトークンが返ってくる→以降はそのトークンでカレンダーの予定を取得できる。

Workers側のコードはClaude Codeが書いてくれる。自分でやるのはGoogle Cloud Consoleでの設定だけ。ここはブラウザ上での操作が必要で、画面が英語なのもあってちょっと大変。ただ、一度設定すれば終わりだし、Claude（チャット）に「Google Cloud ConsoleでOAuth設定する手順を教えて」と聞けばステップごとにガイドしてくれる。

一度つながってしまえば、あとはレベル1と同じ感覚。fetchでAPIを叩いてデータを取るだけ。Life OSのGoogleカレンダー連携もこのレベルで実装した。

### レベル3：APIが有料または制限が厳しい（覚悟がいる）

X（Twitter）のAPI（自動投稿）がこれにあたる。Basicプランで月$100。個人の趣味用途で払う金額じゃない。ただ、前述のようにインテントURL方式なら無料で済む。

レベル3のAPIは本当に必要になるまで手を出さなくていい。たいていの場合、レベル1や2の回避策がある。

| LINE | 通知送信 | 2 | 無料枠あり |
| Slack | メッセージ送信 | 2 | 無料 |

連携のコツとして、Claude Codeに丸投げする前にClaudeチャットで「このAPIとの連携方法を教えて」と全体像を把握しておくと、実装がスムーズに進む。APIのドキュメントURLを教えてあげると精度も上がる。

## よくあるつまづきポイント

バイブコーディングは魔法じゃないので、つまづくこともある。自分が経験したものをいくつか。

**Claude Codeが意図と違うものを作った。**
これは指示が曖昧なことがほとんど。「いい感じにして」だと、Claude Codeの「いい感じ」と自分の「いい感じ」がずれる。色、サイズ、配置、挙動、できるだけ具体的に言う。参考画像があれば見せる。それでもずれたら「ここが違う、こうして」と修正を繰り返せばいい。

**Cloudflareのデプロイでエラーが出る。**
wrangler.toml（Cloudflareの設定ファイル）の記述ミスが多い。D1のdatabase_idが間違ってるとか、環境変数名がコード側と一致してないとか。エラーメッセージをClaudeに貼り付ければだいたい解決する。

**D1のテーブルを変更したい。**
アプリを使い込んでるうちにカラムを足したくなる。D1はSQLiteベースなのでALTER TABLEで追加できるが、手順をミスるとデータが消える可能性がある。Claude Codeに「この列を追加して、既存データは保持して」と伝えれば、安全なマイグレーションを書いてくれる。

**環境変数の設定忘れ。**
ローカルでは動くのに本番でエラーになる、の原因ナンバーワン。Cloudflareダッシュボードで環境変数を設定し忘れてるだけ。APIキーを追加したら、コードの修正と一緒に環境変数も確認する習慣をつける。
